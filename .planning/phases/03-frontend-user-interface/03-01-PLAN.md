    ## HTML Structure

    Create an HTML5 document with `lang="en"`, `meta charset="UTF-8"`, `meta name="viewport" content="width=device-width, initial-scale=1.0"`, and title "YouTube Video Downloader". The body contains these sections (each toggled via a `.hidden` CSS class):

    - **Loading indicator** (`id="loading"`, class `hidden`): A `.spinner` div and `<p>"Fetching video information..."`.
    - **Video preview** (`id="preview"`, class `hidden`): A `.video-card` containing an `<img id="thumbnail">` (alt "Video thumbnail") and `.video-info` with `<h2 id="video-title">` and a `<ul class="meta">` with `<li id="video-duration">`.     ## JS Behavior Requirements

    Write vanilla JavaScript. No libraries. Key behaviors:

    ### URL Auto-Fetch (CORE-01)
    1. Listen for `input` event on `#url-input` with 500ms debounce.
    2. On debounce fire: get trimmed URL value. If it looks like a potential YouTube URL (contains "youtube.com/watch" or "youtu.be"), enable `#fetch-btn` and call `fetchVideoInfo(url)`. If URL is empty or clearly not a YouTube link, disable the button and hide preview.
    3. `fetchVideoInfo(url)`: POST /info with `{"url": url}`. On success (200): populate preview card, populate format dropdown. On error (non-200): show error message from API response `.error` field.
    4. While fetching: show loading spinner, hide preview and error.

    ### Video Preview Display
    1. Set `#thumbnail.src` to `thumbnail_url` from API response. Set `#video-title.textContent` to `title`.
    2. Format `duration` (seconds) into human-readable `"M:SS"` or `"H:MM:SS"` — display as `"Duration: {formatted}"`.
    3. IMPORTANT: Use `textContent` (NOT `innerHTML`) for all text derived from API responses (title, duration, format labels, error messages). This prevents XSS via malicious YouTube metadata per T-03-01.
    4. Show preview section (remove `hidden` class).

    ### Format Selection (CORE-03)
    1. Clear and populate `#format-select` from the `formats` array in API response.
    2. For each format: `<option value="{format_id}">{quality} — {file_size_human} ({ext})</option>`.
    3. `file_size_human`: Format bytes to human-readable: `< 1024 → "N/A"`, else `filesize / 1024 / 1024` → `"X.X MB"` or `"X.X GB"`. Use `file_size` from API (may be null — show "Unknown size").
    4. Add a visual label for formats with both `has_video` and `has_audio` (most user-friendly).
    5. Enable `#download-btn` when a format is selected.

    ### Download with Progress (CORE-03, CORE-05)
    1. On `#download-btn` click: get URL from input, format_id from select.
    2. Disable button and input, change button text to "Starting download...".
    3. POST /download/start with `{"url": url, "format_id": format_id}`.
    4. On success: get `download_id` from response. Show progress section, hide preview. Start polling.
    5. Polling: `setInterval` every 500ms, call GET /download/progress/{download_id}.
    6. Update progress bar:
       - `#progress-fill.style.width` = `percent + "%"`
       - `#progress-percent.textContent` = `percent.toFixed(1) + "%"`
       - `#progress-speed.textContent` = format speed bytes/sec to human-readable: `< 1024 → "X B/s"`, `< 1024*1024 → "X.X KiB/s"`, else `"X.X MiB/s"`. If null/0, show "—".
       - `#progress-eta.textContent` = format ETA seconds to `"M:SS"` or `"H:MM:SS"`, or "—" if null/0.
       - `#progress-size.textContent` = `formatBytes(downloaded_bytes) + " / " + formatBytes(total_bytes || 0)`.
    7. On status "completed": stop polling, set progress bar to 100%, show "Download complete!" message. After 500ms delay, set `window.location = "/download/result/" + download_id` to trigger browser file download.
    8. On status "error": stop polling, hide progress section, show error message from `error_message`.

    ### Error Display (Success Criterion 5)
    1. `showError(message)`: Set `#error.textContent` to `message`, remove `hidden` class.
    2. `hideError()`: Add `hidden` class, clear text.
    3. Call `showError()` on:
       - Fetch failure to /info (non-200 response — use `.error` field from JSON body)
       - Fetch failure to /download/start (non-200 response — use `.error` field)
       - Progress polling returning status "error" (use `.error_message` field)
       - Network errors (try/catch on fetch — show "Network error. Please check your connection.")
    4. Ensure error messages from the API are displayed verbatim (they're already user-friendly from D-14 mapping).

    ### URL Input Button (alternative trigger)
    1. The `#fetch-btn` provides an explicit "Get Info" button (disabled by default, enabled only when valid-looking URL is in the input).
    2. Click handler: same as debounce handler — calls `fetchVideoInfo(url)`.

    ### Reset handling
    1. After a download completes or errors: allow user to paste a new URL and start over.
    2. Enable input and button, reset preview/progress/error sections.

    **IMPORTANT implementation constraints**:
    - No `innerHTML` for any user/API-derived content — always use `textContent` or `setAttribute` (prevention: T-03-01 XSS mitigation).
    - No inline event handlers in HTML (no `onclick="..."`) — use `addEventListener` in JS.
    - No external dependencies — no CDN links for CSS or JS.
    - The page must work without JavaScript errors on modern browsers (Chrome/Firefox/Safari/Edge, last 2 major versions).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Flask API | Untrusted user input (URL string, format_id) crosses from HTML form to backend endpoints |
| Flask API → Browser | Video metadata (title, thumbnail URL, duration) from YouTube crosses via yt-dlp to the frontend DOM |
| Server-side download → Browser | Downloaded video file crosses from server temp dir to browser via streaming response |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-01 | Tampering | Frontend DOM rendering of video metadata | medium | mitigate | All API-derived text (title, duration, format labels, error messages) rendered via `textContent` or `setAttribute`, never `innerHTML`. JS action enforces this with assertion. |
| T-03-02 | Denial of Service | POST /download/start spawning threads | low | mitigate | Frontend disables download button during active download to prevent rapid duplicate clicks. Server validates each request independently. Background threads are daemon=True (auto-terminate on shutdown). |
| T-03-03 | Information Disclosure | GET /download/progress/ and /download/result/ | low | mitigate | download_id generated via `uuid.uuid4()` (128-bit random UUID4, cryptographically unpredictable). Unknown IDs return 404. |
| T-03-04 | Tampering | Content-Disposition filename from YouTube title | medium | mitigate | Safe filename computed using same regex sanitization as existing /download endpoint (line 492 in app.py: `re.sub(r"[^\w\s.-]", "", ...)`) before setting Content-Disposition header in /download/result. |
| T-03-05 | Tampering | Unauthenticated access to any download_id | low | mitigate | download result endpoint checks status==completed before serving file. Unknown download_ids return 404. Progress endpoint only returns safe subset of state (no file_path). |
| T-03-SC | Tampering | pip install of new packages | low | accept | No new Python dependencies introduced in this phase. Existing flask and yt-dlp remain. No package install required. |
</threat_model>
